## 1. Executive Summary
- Purpose: Provide an automated visual underwriting tool that analyzes uploaded property images (interior + exterior), scores properties on lifestyle/maintenance indicators, and surfaces actionable reports for underwriting and analytics.
- Primary users: Admins (manage users, outbox), Analysts/Underwriters (view results, export reports), Field users (upload images to generate a report).

## 2. Goals & Objectives
- Accurately analyze images to derive structured attributes (exterior, interior, assets).
- Produce a normalized risk score and a detailed breakdown by category.
- Provide a web UI for uploads, progress, results, and admin management.
- Support offline email delivery via `backend/outbox` when SMTP is not configured.

## 3. Scope (In-Scope)
- Image upload endpoint and UI pages to submit property images.
- Background processing pipeline that calls an ML/AI module (`gemini_engine.run_full_pipeline`) to analyze images and compute scores.
- Result view with scoring breakdown, risk level, and downloadable activity reports (PDF/TXT).
- Admin: user management, SMTP/outbox management, resend capability, and SMTP status testing endpoints.
- Persistence of usage and event logs in embedded SQLite (`backend/lifestyle_index.db`).
- Generation and storage of `.eml` files when SMTP is not configured.
- JSON export (pre-built JSON files in `json_output/` indicate example outputs per state).

## 4. Out of Scope
- Replacing the AI model; the project integrates with Google Gemini (via `GEMINI_API_KEY`) and expects that model to provide analysis.
- External user authentication providers (OAuth) — user management is local/admin-driven.
- High-availability deployment and autoscaling design (covered in non-functional recommendations only).

## 5. Stakeholders
- Product Owner: internal underwriting team
- Admins: system administrators who create users and manage outbox
- Analysts/Underwriters: review scoring and export reports
- Field Operators: upload images for a household/property
- Dev/Ops: maintain backend, manage API keys and SMTP

## 6. User Roles & Permissions
- Admin: add/remove users, view/resend outbox, trigger test SMTP, view usage metrics
- User (analyst/field): upload images, view processing status, view/download results and reports
- System (background worker): process uploads and attempt email retries

## 7. Functional Requirements (High Level)
1. Upload Images
   - Endpoint/UI to upload multiple images per session; stored under `backend/uploads/<session_id>/`.
   - Validate allowed filetypes (.jpg, .jpeg, .png, .webp).
2. Background Processing
   - A background worker will run `process_images(session_id, session_path)` which invokes `gemini_engine.run_full_pipeline(image_paths)`.
   - Status tracked in `PROCESS_STATUS` and results in `RESULT_STORE` (session-scoped memory + persisted events in DB).
3. Scoring & Output
   - Raw analysis (attributes) and numeric category scores returned by the engine.
   - Compute normalized score and map to `Low/Medium/High Risk`.
   - Provide category breakdown (interior/exterior/asset scores) and image links.
4. Reports & Exports
   - Generate downloadable user activity reports (PDF via reportlab or fallback to TXT).
   - Provide JSON outputs (examples located in `json_output/`).
5. Admin Emailing & Outbox
   - If SMTP configured via `.env`, send emails (with attachments) via SMTP.
   - If SMTP missing, save `.eml` files to `backend/outbox` and support manual/resend flows.
6. Logging & Metrics
   - Record events and API usage in SQLite tables `events` and `api_usage` for quota and auditing.

## 8. Data Inputs & Outputs
- Inputs:
  - Uploaded images (user-provided) under `backend/uploads/<session_id>/`.
  - Environment variables in `.env`: `GEMINI_API_KEY`, optional SMTP settings, `ADMIN_EMAIL`, `SECRET_KEY`.
- Outputs:
  - Processed result objects in-memory (`RESULT_STORE`) and rendered pages: `result.html`, `processing.html`.
  - Example JSON reports in `json_output/` (per-state sample outputs).
  - `.eml` files in `backend/outbox/` if SMTP not configured.
  - Activity and usage entries in `backend/lifestyle_index.db`.

## 9. User Workflows (Primary)
1. User uploads images via UI (or API).
2. System creates a `session_id`, stores images in `uploads/<session_id>/`, returns processing page.
3. Background thread invokes `run_full_pipeline(...)` (Gemini), computes scores and analysis.
4. System updates `PROCESS_STATUS` then the UI `result.html` fetches `RESULT_STORE[session_id]` and displays output.
5. Admins may view `outbox` and resend saved emails once SMTP is configured.

## 10. Non-Functional Requirements
- Performance: Typical session (3-6 images) processed within acceptable latency (depends on Gemini latency). Background processing avoids blocking UI.
- Scalability: Current implementation uses in-memory result store and a single-process worker; plan to migrate to queue-based workers (RabbitMQ/Celery or cloud queues) for scale.
- Security: Secrets must be stored in `.env` (not in repo). Session middleware uses `SECRET_KEY`. Uploaded files must be validated.
- Reliability: Save outgoing emails to disk when SMTP fails; retry worker to resend `.eml` files.
- Privacy: Images and results are sensitive; ensure access control on UI and secure storage (HTTPS, ACLs) in production.

## 11. Data Model (summary)
- SQLite `events` table: logs user events (login, upload, report_generated).
- SQLite `api_usage` table: token usage (input/output/total) per user session.
- Filesystem: `backend/uploads/<session_id>/` (images), `backend/outbox/` (.eml), `json_output/*.json` example exports.

## 12. Acceptance Criteria
- A user can upload images and receive a scored result (risk level + breakdown) in the UI.
- The background pipeline must update status from `PROCESSING` → `DONE` or `ERROR` and store results accessible by session id.
- If SMTP not configured, outgoing emails are saved to `backend/outbox` and can be resent after configuration.
- Events and API usage are recorded in `lifestyle_index.db`.

## 13. Assumptions & Constraints
- The scoring engine is provided by Google Gemini via `gemini_engine.py` and requires a valid `GEMINI_API_KEY`.
- Current worker strategy is single-process and in-memory; memory loss on restart is acceptable for now but should be addressed for production.
- User authentication is handled via lightweight session middleware; no external SSO is implemented.

## 14. Risks & Mitigations
- API Key exposure or misuse: enforce environment-based keys and rotate periodically.
- Model drift or incorrect labels: maintain a mapping review process and unit tests for `SCORING_CONFIG` values.
- Scalability bottlenecks: plan migration to queue workers and persistent result storage (DB/Blob store).
